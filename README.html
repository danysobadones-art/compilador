🔎 ¿Qué es un compilador?

Un compilador es un programa que traduce código fuente escrito en un lenguaje de alto nivel (como C++) a una representación intermedia o a código máquina, permitiendo que el programa pueda ejecutarse en un sistema.

El proceso clásico de compilación consta de varias etapas:

  • Análisis Léxico
  • Análisis Sintáctico
  • Análisis Semántico
  • Generación de Código Intermedio
  • Optimización
  • Generación de Código Objeto
TECHNOCOMPILER implementa este ciclo completo bajo una arquitectura modular, permitiendo analizar, transformar y optimizar código C++ de manera estructurada y segura.

🧠 Metodología de Desarrollo

Se utilizó una metodología incremental y modular.

¿Por qué Incremental?

Porque el proceso de compilación depende de que cada etapa funcione correctamente antes de pasar a la siguiente.

¿Por qué Modular?

Porque cada etapa del compilador cumple una responsabilidad específica y puede evolucionar de forma independiente.

ETAPA 1 — Análisis Léxico

🔹 Implementación del Análisis Léxico

¿Qué significa?

El análisis léxico convierte el texto del programa en tokens — las unidades básicas reconocibles del lenguaje.

¿Por qué se usó?

Porque ningún compilador puede analizar estructuras sin primero dividir el código en unidades básicas reconocibles.

¿Cómo se usó?

Se implementaron expresiones regulares en lexico.py para reconocer:

  • Identificadores
  • Operadores
  • Palabras clave
  • Símbolos especiales

Se validó generando listas de tokens antes de continuar.

ETAPA 2 — Análisis Sintáctico

🔹 Construcción del Parser

¿Qué significa?

El parser organiza los tokens según la gramática del lenguaje.

¿Por qué se usó?

Para verificar que el código tenga estructura válida antes de intentar optimizarlo.

¿Cómo se usó?

En parser.py se:

  • Detectaron asignaciones
  • Identificaron estructuras while y do-while
  • Construyeron nodos del AST utilizando nodo.py
Cada estructura válida se convertía en un nodo del árbol.
ETAPA 3 — Construcción del AST

🔹 Árbol de Sintaxis Abstracta

¿Qué significa?

El AST (Árbol de Sintaxis Abstracta) representa el programa en forma jerárquica.

¿Por qué se usó?

Porque permite manipular el programa de forma estructurada sin depender del texto original.

¿Cómo se usó?

Cada instrucción se representó como objeto nodo con atributos específicos.

Esto permitió generar el IR (Código Intermedio) en la siguiente etapa de forma segura y estructurada.
ETAPA 4 — Generación de Código Intermedio (IR)

🔹 Representación Intermedia

¿Qué significa?

El IR es una representación abstracta del programa que facilita optimizaciones.

¿Por qué se usó?

Porque optimizar directamente el código fuente es complejo y riesgoso. El IR permite aplicar transformaciones seguras antes de reconstruir el código.

¿Cómo se usó?

En generador_ir.py se:

  • Crearon labels automáticos (L0, L1, L2…)
  • Transformaron bucles en saltos condicionales
if !(condicion) goto L1
   ...
goto L0
ETAPA 5 — Optimización de Código

🔹 Implementación del Optimizador

¿Qué significa?

Reducir operaciones innecesarias manteniendo el mismo resultado lógico.

¿Por qué Constant Folding?

Porque es una optimización segura, controlada y demostrable académicamente.

¿Cómo se usó?

En optimizador.py se detectaron expresiones constantes, se evaluaron en tiempo de compilación y se reemplazaron por su valor final:

ANTES
a = 2 + 3;
DESPUÉS
a = 5;
ETAPA 6 — Generación de Código / Síntesis

🔹 Reconstrucción del Código

¿Qué significa?

Volver a generar código C++ legible a partir del IR optimizado.

¿Por qué se usó?

Para demostrar el ciclo completo de transformación: del código fuente al código optimizado.

¿Cómo se usó?

En reconstructor.py:

  • Se reemplazaron asignaciones optimizadas
  • Se mantuvo intacta la estructura no optimizable
📄 optimizado_nombrearchivo.cpp
ETAPA 7 — Arquitectura del Sistema

🔹 Modularización Final

¿Qué significa?

Separar responsabilidades por archivos independientes.

¿Por qué se usó?

  • Integración futura de nuevas optimizaciones
